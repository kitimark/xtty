# `xtty` Research

Research library on terminal emulators — competitive landscape, internals, and design implications for building a macOS-native terminal (`xtty`).

> **Provenance:** Generated 2026-06-27 from a 39-agent research workflow (10-terminal survey + adversarial fact-check + 8 architecture deep-dives + synthesis), 281 web lookups. Claims marked surprising/marketing-flavored were independently fact-checked (see [fact-checks](../03-analysis/fact-checks.md)).

### 01 — Terminals (deep dives)
- [iTerm2](01-terminals/iterm2.md) — The most feature-complete, deeply macOS-integrated terminal for power users, with best-in-class tmux control-mode integration and extensive scripting/automation.
- [Apple Terminal.app (macOS built-in)](01-terminals/apple-terminal.md) — The zero-install, Apple-supported default terminal that ships with macOS — maximally reliable and integrated, but feature-conservative compared to third-party emulators.
- [Ghostty](01-terminals/ghostty.md) — A terminal that refuses to trade off speed, features, or native-platform feel simultaneously, built on a reusable C-ABI core (libghostty) with a true Metal/SwiftUI macOS app rather than a cross-platform widget toolkit.
- [kitty](01-terminals/kitty.md) — A "terminal as a platform": GPU-accelerated, highly hackable, and the originator of the kitty graphics protocol plus the kittens extension framework now influencing the broader terminal ecosystem.
- [Alacritty](01-terminals/alacritty.md) — A deliberately minimalist, GPU-accelerated terminal that does one thing — render text fast — and offloads tabs/splits/multiplexing to tools like tmux.
- [WezTerm](01-terminals/wezterm.md) — A fully cross-platform, Rust-based terminal that bundles a tmux-style multiplexer and deep Lua scripting into one GPU-accelerated app.
- [Warp](01-terminals/warp.md) — An AI-native, blocks-based terminal that has evolved into a full "agentic development environment" — combining a polished native terminal with multi-agent orchestration, rather than just a fast text grid.
- [Rio Terminal](01-terminals/rio.md) — A Rust, GPU-accelerated terminal with a custom Sugarloaf/wgpu renderer that targets desktops, FreeBSD, and the browser (WebAssembly) from one codebase, with an emerging embeddable engine (librio) and a native-Swift macOS frontend (Super Rio).
- [Wave Terminal](01-terminals/wave.md) — A block/widget-based terminal that pulls files, previews, an editor, a web browser, and AI into one workspace to eliminate context switching, rather than chasing raw rendering speed
- [Hyper](01-terminals/hyper.md) — A fully web-tech terminal (Electron + React + Redux + xterm.js) whose entire UI and plugin/theme system is built and customized with HTML/CSS/JavaScript via npm.

Adversarial verification overturned 4 carried-in claims (fork-free Tier-0 copy silently corrupts; the fork is 2 accessors; Tier-2 isn't fragile; the naive `linesTop`-decrease invalidation is broken) → corrected invalidation = `invalidate-all-on-sizeChanged` + `liveTop` high-water-drop + normal-buffer pin + alt-screen gate. **2026-06-29 P4b-3 addendum:** external research (WebSearch + an iTerm2 `/tmp` source clone) **overturns D3's premise** — iTerm2 (keyboard-first) *does* ship a clickable command list (Toolbelt: single-click→scroll-to-mark, shell-integration-gated, a secondary drawer); its `VT100ScreenMark` invalidation (eager per-mark recompute on resize + eager trim GC) is the eager twin of xtty's leaner lazy epoch-bump (same stale-click-no-op UX, but we skip iTerm2's width-baked-in full recompute). P4b-3 reframed from "deferred" → a cheap, prior-art-validated *secondary* convenience (keyboard stays primary); decisions-to-lock listed.
- [SwiftTerm fork vs patch-in-repo](03-analysis/swiftterm-fork-vs-patch-strategy.md) — how to inject the P4b-2 accessors **without creating a fork repo**: a Playwright-style patch-at-build doesn't map onto SwiftPM (SPM owns the checkout), but the spirit does via a **local-path dep** (submodule/vendor); and because the change is add-only one file, the "patch" is a drop-in. Plus the **defer-via-injectable-shim** analysis (Int-only seam → ~2-line light-up, fake-tested happy path). **Follow-up:** Playwright actually **gitignores the upstream checkout** (`/checkout`) rather than submoduling it — *a submodule is the wrong primitive for a patched dep* — so the cleanest no-repo mechanism is a **gitignored clone + bootstrap-enforced pin** (B′), with a throwaway **fork** cleaner still if a repo is acceptable; the whole mechanism is temporary (retired when the accessors land upstream)
- [P6 file/diff decisions](03-analysis/p6-file-diff-decisions.md) — two-pass (web + deep-read of zed/vscode/lazygit/gitui/delta/waveterm source) decisions for the file/diff view: ship **P6a**, a read-only **git-review panel** (changed-files grouped Changes/Untracked/Conflicts → click-to-diff, plain unified SwiftUI diff, right-edge collapsible panel) via **git shell-out** (not libgit2); refresh on OSC-133 command-finish **+ a ~5 s gated poll backstop** (the agent-mid-command gap both git TUIs poll for); **porcelain v1 `-z`** (not v2); defer the file tree (P6b) + staging. Reading real source corrected 4 pass-1 calls. **P6a+ addendum (2026-06-29):** decides intra-line diff emphasis for `add-git-review-polish` — **token-level** DP (not trim-only, not char-level), zed/delta-gated, `Text(AttributedString)` per-run background (macOS 12+, no fallback), + a one-line pause-during-own-git poll guard. **P6b addendum (2026-06-29):** narrows P6b to **Scope A** — a flat↔tree toggle over the *changed* files (pure view-transform, on-mission for big agent changesets) — and **rejects Scope B**, the full project file-tree browser, as off-mission IDE-creep; recommends deferring the build until after P7-measure
- [P7 measurement methodology & scope](03-analysis/p7-measurement-methodology.md) — explore-phase plan for the **P7 measure-and-decide gate**: the bar is undefined (M1/M4 qualitative) → set it **relative** to installed comparators (Warp/iTerm2/Terminal.app); latency via a **fork-free in-process screen-capture probe** (inject `CGEvent` → poll pixels; renderer-agnostic, exact A/B delta) over a per-renderer SwiftTerm patch; memory via `task_info` under fixed scenarios; a base-only `renderer = coregraphics|metal` key for rebuild-free A/B; **defer distribution** (0 codesigning identities on the dev machine); split P7 into **P7a `add-latency-memory-harness`** (gate-critical, fork-free) → P7b decision → P7c memory pass. **2026-06-29 apply addendum:** building + running P7a showed **memory is lean & trustworthy** (idle ~68 MB footprint, 0 % idle CPU, ≤135 MB saturated — M1 met) but the **screenshot-polling latency probe is too coarse** (capture ~20 ms > the signal → can't resolve key-to-photon; real probe + CG-vs-Metal verdict = P7b); plus an **operational signing finding** (ad-hoc rebuilds re-prompt for Screen Recording → a stable self-signed `xtty-dev` cert via `scripts/create-signing-cert.sh` + `XTTY_SIGN_IDENTITY` makes the TCC grant persist). **2026-06-29 P7b addendum** (`/opsx:explore p7b`, 14-agent research workflow + SwiftTerm-checkout deep-read): the full **trustworthy-probe methodology** — replace polling with a continuous **`SCStream` reading per-frame `displayTime`** (the next `.complete` frame after t0 *is* the keystroke; on-glass, fork-free, ~90 % P7a reuse), a load-bearing **clock-unit fix** (`displayTime` is mach *ticks* not ns — normalize both ends via `CMClockMakeHostTimeFromSystemUnits`), a **startup epoch-calibration gate**, and the **headline reframe**: both renderers sit behind a **shared ~16.67 ms output-coalescing throttle** (`queuePendingDisplay`), so the renderer A/B is a **wash** — the throttle, not the renderer, is xtty's latency lever. **GO, expected verdict: keep CoreGraphics, skip Phase 8**; comparators = Terminal.app-first stretch (iTerm2 Secure-Event-Input blocks injection); no hardware rig needed (the omitted tail cancels in the delta). **2026-06-29 P7b apply result** (`add-trustworthy-latency-probe`, implemented & benched 3×/renderer): the probe works — **calibration passed** (offset <4 ms → absolute numbers trustworthy on macOS 26.2) and latency is resolved (P7a floored at ~54 ms). **Verdict CONFIRMED: keep CoreGraphics, skip Phase 8** — CG faster median (~31 vs ~33 ms), much tighter tail (p99 ~40 vs ~50–120 ms), leaner memory (~55 vs ~62 MB idle); the shared throttle dominates the median, the probe additionally exposed Metal's worse tail. The renderer question is settled; P7c (leaks) + distribution remain → see the dedicated P7c doc next
- [P7c leak / retain-cycle audit](03-analysis/p7c-leak-retain-audit.md) — methodology & scope for the Phase-7 **leak/retain pass** (`/opsx:explore p7c` + a 14-agent workflow, 2026-06-29). **Headline: no leak to hunt** — both explore-flagged cycles (`WindowController⇄GitReviewController`, `PaneController⇄LinkRoutingTerminalDelegate`) **refuted** (all closures `[weak self]`, delegates `weak`, `GitReviewTarget` a transient value type); App-layer sweep found zero cycles, only 2 cosmetic nits. So P7c reframes to **prove + guard + profile-SwiftTerm**: a DEBUG **live-instance census** (`liveCount` init`++`/deinit`--` on 6 lifecycle types → `liveInstanceCounts` in the state dump → churn XCUITest asserts return-to-baseline = the gated net), weak-sentinel teardown for in-process `XttyCore` types, a one-time CLI `leaks`/`xctrace`/`vmmap` pass (no special entitlements; **not** CI-gated — false-positive-prone) targeting the one unnameable risk: SwiftTerm's **unbounded glyph/font caches** + incomplete renderer `deinit`. Footprint-after-churn stays **report-only** (too noisy). **Diverges from the raw synthesis:** drop the proposed `max-memory-mb` config key (knob with no runtime behavior, contradicts the noisy-footprint finding). Recommended change: **`add-lifecycle-census`** (fork-free); gates nothing; distribution stays deferred
- [Distribution: Developer ID + Hardened Runtime + notarization](03-analysis/distribution-signing-research.md) — explore-phase finding for the deferred **P7 distribution slice** (`/opsx:explore distribution`, 7-agent workflow, 2026-06-30). **Headline: technically green, gated only on ~$99/yr procurement.** The crux is a non-issue — **Hardened Runtime does NOT break a terminal that execs unsigned user binaries** (HR constrains only xtty's own process image; child shells/`$EDITOR`/CLIs get independent AMFI evaluation; ZDOTDIR injection is a *child*-env write, unrelated to `allow-dyld-environment-variables`); proven by iTerm2/Ghostty/WezTerm all notarized under HR with zero `allow-unsigned-executable-memory`. **The HR-exception entitlement set is empty** — just flip `ENABLE_HARDENED_RUNTIME=YES`, keep sandbox-off, sign with a real Developer ID, ship a Release archive (Ghostty, the closest analog, does exactly this). **Nothing xtty built breaks** — 5/6 surfaces SAFE, the one change is build-config not code; SwiftTerm static-links (no separate framework to notarize). Pipeline (`notarytool` era, `altool` dead): archive → inside-out `codesign -o runtime --timestamp` → `notarytool submit --wait` → `stapler staple` → `spctl`/verify gate, all scriptable as `make dist`. A real Developer ID **subsumes** the `xtty-dev` TCC workaround for signed builds. Recommended change: **`add-distribution-signing`** (`build-workflow` + `verification-harness` deltas); **blocked on the $99/yr decision only the owner can make** — captured, not proposed. Critic-flagged verify-before-spending items (SwiftTerm same-team re-sign, static-link, resource-access TCC set in/out) in §8. **§9 addendum (2026-06-30 — `/opsx:explore` "is brew.sh possible?", 5-agent workflow):** yes via a Homebrew **Cask** — but it's a channel layered *on top of* notarization, **not a bypass** (cask quarantines by default → Gatekeeper still adjudicates; the `--no-quarantine` escape is gone in Homebrew 6.0.5; Homebrew is *disabling* un-notarized casks Sept-2026, e.g. Alacritty's). **Own tap first** (`kitimark/homebrew-xtty`, no notability gate), official tap later. **Tap cost vs app cost:** the own tap is **free** (the $99 is the app's Gatekeeper gate, not the tap) — an ad-hoc-own-tap path is **$0** but every user hits the "unverified developer" wall + manual "Open Anyway" (Alacritty-style); only the $99 (notarization) buys a friction-free first launch. A **separate** thin follow-on change `add-homebrew-distribution` downstream of `add-distribution-signing`; comparator casks read live via local `brew`. **§10 addendum (2026-06-30 — `/opsx:explore` "distribute with 0 cost", 5-agent workflow):** **$0 distribution already works, split by *who compiles the binary*** — a **locally-compiled** app dodges Gatekeeper entirely (verified: `xattr -l` on the repo's own Release build shows no `com.apple.quarantine`), so **source-build (`make`) is the canonical $0 + clean path and xtty is already there**; a **downloaded** ad-hoc prebuilt is $0 but costs a one-time "Open Anyway"/`xattr` per user (right-click→Open gone since macOS 15); a clean download = the $99 `add-distribution-signing`. Free Apple ID can't help distribution; Homebrew formula-from-source viable-but-a-rework (deferred). Maps to a README install section + small `build-workflow` delta, distinct from the $99 change
- [CI/CD on GitHub Actions](03-analysis/github-actions-ci-cd.md) — explore-phase finding for a **`add-ci-pipeline`** tooling change (`/opsx:explore` "can we run a pipeline in GitHub Actions?", 6-agent workflow, 2026-06-30; the repo `git@github.com:kitimark/xtty.git` exists but the local clone has no remote + no `.github/` yet). **Verdict: yes — $0, secret-free CI/CD on GitHub-hosted `macos-26` runners today** (the ad-hoc "Sign to Run Locally" default needs no Apple secrets; standard runners are free+unlimited on public repos). Design: two jobs on push/PR — a **required** fast headless `test-core` (`swift test XttyCore`) + a **best-effort** `build-and-test` (XcodeGen → `xcodebuild test`, retry-tolerant); highest-value cache = `external/SwiftTerm` keyed on the pin+patch hash. **Two unknowns to design defensively around** (critic-corrected — the synthesis oversold the first): ❓ the **Metal toolchain on the runner** is a genuine 2-2 facet split (likely preinstalled on macos-26 release Xcode ≥26, but excluded for RCs) → keep an idempotent `xcrun -f metal || sudo xcodebuild -downloadComponent MetalToolchain` guard + verify on first run; ❓ **XCUITest drive-path** (`typeText`/`click`) reliability on hosted runners is unproven → gate on the headless job, start the GUI job non-blocking. Release = a separate tag-triggered job ($0 ad-hoc DMG → `gh release` now; the $99 notarized seam deferred to `add-release-notarization`). Maps to a **`build-workflow` spec delta** (no `verification-harness` delta — CI invokes existing tests, adds no observable behavior). Companion to the distribution doc above. **§9 addendum (2026-06-30 — first CI run `28425122861`):** implemented as `add-ci-pipeline`, pushed, run — **both unknowns resolved favorably.** Metal is **preinstalled** on the release-default Xcode 26.5 (guard a no-op); **`test-core` green in 1m6s** (the gate works); the predicted **XCUITest-on-hosted-runner risk is REFUTED** — **34/41 pass** (drive path, Cmd+V, zsh injection, grid+state dumps all work). **7 deterministic CI-env failures** (focus/activation cluster + clipboard + truecolor/locale) → the non-blocking job correctly *surfaced* them; hardening is follow-up. `add-ci-pipeline` left **open**. **§10 addendum (2026-06-30 — `harden-xcuitests-for-ci` diagnosis + an 8-repo per-project verification):** **"headless" was the wrong word** (corrected — GitHub runners have a real auto-login Aqua session; the true issue is a native app can't *reliably* own the *shared* host session — key-window/menu-bar/clipboard — when XCUITest fires; **unreliable, not impossible**). The "Simulators work ⇒ macOS-native UI testing works" inference is **invalid** (iOS-sim = isolated window server). OSS field: **1 robust (DuckDuckGo apple-browsers — menu/clipboard XCUITests green on hosted `macos-26`, the genuine counter-example) · 1 flaky (Maccy) · 6 avoid · 1 explicit opt-out (Ghostty, a native terminal, excludes XCUITests from CI)**. Refined plan = **hybrid**: keep the env-trigger/grid-dump side channel (the only path for xtty's custom-drawn, no-a11y content) + adopt DDG's primitives (a11y-IDs on NSMenuItems + click-first + `typeKey`) for the few must-drive-GUI tests + rewrite the focus-on-activate test + keep the GUI job non-blocking. Maps to a `verification-harness` delta **§11 addendum (2026-07-01 — first read of the actual `.xcresult` screenshots/grid-dumps, not just logs):** corrects §9–§10 — authoritative counts are **30/7/1**, the runner shell is **`/bin/bash`** (a missed factor: its zsh-deprecation banner corrupts grids, disables shell-integration, and makes `testFocusTyping…` a **false negative** → *keep* the test, don't rewrite it), truecolor/emoji is **Cmd+V** not a font/locale gap, and xtty **does** own its menu bar; net — the Cmd-key/menu thesis is **screenshot-confirmed** (6/7 failures), **fix the shell first** then scope hardening to that Bucket-B residual. **§12 addendum (2026-07-01 — `silence-bash-deprecation` measured in CI, run `28467944762`):** the fix **works** (banner gone from **0/221** attachments — Bucket A retired) but flips **zero** tests green (still **30/7/1**); the §11f "fix the shell → focus-typing flips green" prediction is **REFUTED** — `testFocusTyping…` still fails because the runner's ~72-char hostname prompt **wraps the typed marker across rows** (not the banner; banner-free run still wraps), so its fix becomes a **wrap-tolerant assertion**, and the residual is a clean **6 Bucket-B + 1 wrap** for `harden-xcuitests-for-ci`. **#1 (the wrap) carved out + implemented as `harden-focus-typing-assertion`** (scoped opt-in wrap-tolerant matcher, test-only, one `verification-harness` requirement). **§13 addendum (2026-07-01 — local `make test` of the churn test #4):** root-caused the Bucket-B churn flake — split-then-immediate-close races the freshly-split shell's **startup**, so `hasForegroundJob` is true (a startup child owns the PTY foreground) and a **confirm-close `NSAlert` blocks the close** → census `final 2`; fix = the churn test opts out via `confirm-close = false` (or waits for shell-ready). Also caught a **multi-monitor dev artifact**: local `.xcresult` recordings capture the **main/external** display while xtty forces itself onto the **built-in** screen, so they're blind to the popup (CI is single-display → unaffected)
